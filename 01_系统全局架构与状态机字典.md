> 系统名称：光明LIMS质检系统  
> 版本：v3.8.8

## 一、项目目录与技术栈总结

### 1.1 项目整体结构

```
project/
├── lims/              # 后端（Spring Boot + 若依框架）
├── lims-app/          # 移动端（UniApp + 钉钉小程序）
└── lims-vue/          # 管理后台前端（Vue + 若依框架）
```

### 1.2 后端技术栈

**基础框架：**
- **核心框架**：Spring Boot 2.x + 若依（RuoYi）快速开发框架
- **Java版本**：JDK 1.8
- **安全框架**：Spring Security 5.7.12 + JWT 0.9.1
- **持久层框架**：MyBatis + PageHelper 分页插件
- **数据库连接池**：Druid 1.2.23
- **API文档**：Swagger 3.0.0
- **工具库**：FastJSON 2.0.43、Apache POI 4.1.2（Excel处理）

**后端模块结构（Maven多模块）：**
```
gmlimsqi/
├── gmlimsqi-admin        # 管理后台启动模块
├── gmlimsqi-business     # ★核心业务模块（质检业务逻辑）
├── gmlimsqi-system       # 系统管理模块（若依自带）
├── gmlimsqi-common       # 公共工具模块
├── gmlimsqi-framework    # 框架配置模块
├── gmlimsqi-generator    # 代码生成器（若依自带，可忽略）
└── gmlimsqi-quartz       # 定时任务模块（若依自带）
```

**关键架构说明：**
- 系统按组织机构类型（dept_type）进行业务隔离：1-总部、2-牧场、3-饲料场、4-检测中心
- 不同业态的质检流程在代码层面物理隔离，互不干扰

### 1.3 移动端技术栈

- **框架**：UniApp（跨平台开发框架）
- **目标平台**：钉钉小程序
- **主要目录**：`lims-app/pages/`

### 1.4 前端技术栈

- **框架**：Vue.js + 若依前端框架
- **UI组件库**：Element UI
- **主要目录**：`lims-vue/src/`

## 二、核心业务目录标注

### 2.1 后端核心业务模块（重点关注）

**位置**：`lims/gmlimsqi-business/src/main/java/com/gmlimsqi/business/`

**⚠️ 重要架构认知：系统按业务主体（使用方）进行物理隔离**

系统将业务主体明确分为三类，对应不同的代码模块和数据表：
1. **【牧场】** - dept_type=2
2. **【饲料厂】** - dept_type=3  
3. **【中心化验室】** - dept_type=4（检测中心）

#### ⭐ 一级核心：牧场/饲料厂业务模块

**这些模块是牧场和饲料厂专属的质检流程，与中心化验室物理隔离**

| 模块包名 | 业务主体 | 业务说明 | 核心表 |
|---------|---------|---------|--------|
| **samplingplan** | 牧场/饲料厂 | 成品/库存/垫料采样计划管理 | `op_finished_product_sampling_plan` |
| **milksamplequalityinspection** | 牧场 | 奶样质检（进厂检验） | `op_milk_sample_quality_inspection` |
| **milktankinspection** | 牧场 | 奶罐检验 | `op_milk_tank_inspection` |
| **inspectionmilktankers** | 牧场 | 奶车检验 | `op_inspection_milk_tankers` |
| **factoryquality** | 牧场/工厂 | 工厂接收奶源后的质检结果 | `op_factory_quality_inspection` |
| **ranch** | 牧场 | 牧场信息、采样样品管理 | `op_sampling_plan_sample` |

**重要实体类**：
- `OpFinishedProductSamplingPlan` - 成品/库存/垫料取样计划（饲料厂）
- `OpMilkSampleQualityInspection` - 奶样质检单（牧场）
- `OpFactoryQualityInspection` - 工厂质检结果（牧场奶源到工厂后）

#### ⭐ 二级核心：中心化验室业务模块（物理隔离）

**⚠️ 注意：`labtest` 模块是【中心化验室】专属，不是全系统通用的化验核心！**

| 模块包名 | 业务主体 | 业务说明 | 核心表 |
|---------|---------|---------|--------|
| **labtest** | 中心化验室 | 委托单管理、样品检测、报告生成（中心化验室的业务逻辑） | `op_blood_entrust_order`、`op_feed_entrust_order`、`op_jczx_*` |

**说明**：
- 前端对应包名：`CentralLaboratory`
- 中心化验室接收外部送检的【血样】【饲料样品】【PCR样品】，走委托单流程
- 委托单状态机：待提交 → 待受理 → 检测中 → 检测完成 → 已审核 → 已发送

**中心化验室实体类**：
- `OpBloodEntrustOrder` - 血样委托单
- `OpFeedEntrustOrder` - 饲料委托单
- `OpJczxBloodResultBase` - 血样检测结果
- `OpJczxFeedResultBase` - 饲料检测结果

#### ⭐ 三级核心：基础数据与支撑模块（跨业态共享）

| 模块包名 | 业务说明 | 用途 |
|---------|---------|------|
| **basicdata** | 基础数据管理 | 检验项目、检验方法、物料标准、仪器配置（所有业态共享）|
| **unquality** | 不合格品管理 | 不合格判定、处理流程 |
| **report** | 报表统计 | 各类质检报表、数据分析 |
| **instrument** | 仪器设备管理 | 设备档案、保养记录 |
| **reagent** | 试剂管理 | 试剂出入库 |
| **consumable** | 耗材管理 | 耗材出入库 |
| **culture** | 培养基管理 | 培养基配制 |
| **disinfection** | 消毒管理 | 消毒记录 |
| **exam** | 人员考试培训 | 化验员培训考核 |

## 二、核心数据模型与状态机

### 2.1 核心业务表总览

系统所有业务表按前缀分类：
- **`op_`** - Operation（业务操作表）：采样计划、质检单、委托单、检测结果等
- **`bs_`** - Base Setting（基础配置表）：物料、项目、仪器、容器等
- **`sys_`** - System（系统表）：若依框架自带的用户、角色、权限等

### 2.2 牧场/饲料厂核心表与状态机

#### 表1：`op_finished_product_sampling_plan` - 成品/库存/垫料取样计划（饲料厂）

**作用**：饲料厂的成品、库存、垫料的采样计划单主表

**关键字段**：
- `sampling_order_number` - 取样单号
- `plan_type` - 计划类型（0:成品，1:库存，2:垫料）
- `status` - 状态
- `dept_id` - 部门ID（饲料厂）

**状态流转**（`sampling_plan_status`）：
```
0 未提交 → 1 未取样 → 2 已取样
           ↓
        3 已作废
```

**子表**：`op_finished_product_sampling_plan_detail` - 计划明细（物料、生产订单号、计划取样份数）

---

#### 表2：`op_milk_sample_quality_inspection` - 奶样质检表（牧场）

**作用**：牧场奶源进厂时的质检记录，包含奶温、脂肪、蛋白、抗生素等指标检测

**关键字段**：
- `milk_sample_quality_inspection_number` - 奶样质检单号
- `milk_source_plan_order_number` - 奶源计划单号（关联奶源系统）
- `license_plate_number` - 车牌号
- `status` - 状态
- `sampler` / `sampling_time` - 取样人/取样时间
- `tester` / `test_time` - 检测人/检测时间
- `reviewer` / `review_time` - 审核人/审核时间
- `milk_temperature` - 奶温
- `fat_percent` - 脂肪%
- `protein_percent` - 蛋白%
- `beta_lactam` - β-内酰胺（抗生素）
- `aflatoxin_m1` - 黄曲霉毒素M1

**状态流转**：
```
0 待取样 → 1 已取样 → 2 已审核
```

**变更日志表**：`op_milk_sample_quality_inspection_change_log` - 记录数据修改历史

---

#### 表3：`op_factory_quality_inspection` - 工厂质检结果表

**作用**：牧场奶源到达工厂后，工厂侧进行的质检结果（非牧场自检）

**关键字段**：
- `milk_source_plan_order_number` - 奶源计划单号（唯一索引）
- `pasture_code` - 牧场编号
- `factory_code` - 工厂编号
- `actual_milk` - 实际奶量(吨)
- `unload_temperature` - 进厂奶温
- `testing_fat` / `testing_protein` - 脂肪/蛋白
- `testing_colonies` - 菌落总数
- `testing_somatic_cell` - 体细胞
- `testing_aflatoxin` - 黄曲霉毒素M1

**说明**：此表数据由工厂侧检测后回传，用于对比牧场侧和工厂侧的检测结果差异

---

#### 表4：`op_milk_tank_inspection` - 奶罐检验表

**作用**：牧场奶罐的质量检验记录

#### 表5：`op_inspection_milk_tankers` - 奶车检验表

**作用**：奶车进厂时的检验记录（铅封号、奶温、感官检查等）

---

### 2.3 中心化验室核心表与状态机

#### 表6：`op_blood_entrust_order` - 血样委托单（中心化验室）

**作用**：外部送检血样的委托单管理

**关键字段**：
- `entrust_order_no` - 送检单号
- `entrust_dept_id` - 委托单位ID
- `send_sample_date` - 送样时间
- `status` - 委托单状态

**状态流转**（`entrust_order_status`）：
```
0 待提交 → 1 待受理 → 2 检测中 → 3 检测完成 → 4 已审核 → 5 已发送
                                          ↓
                                    6 已驳回 / 7 作废
```

**关联表**：
- `op_blood_entrust_order_sample` - 血样样品明细
- `op_blood_entrust_order_item` - 检测项目明细
- `op_jczx_blood_result_base` - 血样检测结果主表
- `op_jczx_blood_result_info` - 血样检测结果明细
- `op_jczx_blood_report_base` - 血样检测报告主表

---

#### 表7：`op_feed_entrust_order` - 饲料委托单（中心化验室）

**作用**：外部送检饲料样品的委托单管理

**状态流转**：同血样委托单（`entrust_order_status`）

**关联表**：
- `op_feed_entrust_order_sample` - 饲料样品明细
- `op_jczx_feed_result_base` - 饲料检测结果主表（化验单）
- `op_jczx_feed_report_base` - 饲料检测报告主表

**饲料化验单状态流转**（`feed_result_status`）：
```
1 开始化验 → 2 化验完成 → 3 校对完成 → 4 审核完成
               ↓
           5 作废 / 6 退回
```

---

#### 表8：`op_pcr_entrust_order` - PCR委托单（中心化验室）

**作用**：PCR检测（聚合酶链式反应）样品的委托单管理

**关键字段**：
- `entrust_order_no` - 送检单号
- `entrust_dept_id` - 委托单位ID
- `status` - 委托单状态

**状态流转**：同血样委托单（`entrust_order_status`）

**关联表**：
- `op_pcr_entrust_order_sample` - PCR样品明细
- `op_pcr_entrust_order_item` - PCR检测项目明细
- `op_jczx_pcr_result_base` - PCR检测结果主表
- `op_jczx_pcr_result_info` - PCR检测结果明细

**PCR化验单状态流转**（`pcr_result_status`）：
```
1 开始化验 → 2 化验完成 → 3 审核完成
               ↓
           4 作废
```

**说明**：PCR检测主要用于疫病检测（如五联、七联、八联等）

---

#### 表9：`op_milk_filling_order` - 装奶单（牧场奶源外运）

**作用**：牧场奶源装车外运的核心单据，记录装奶信息和奶仓分配

**关键字段**：
- `milk_filling_order_number` - 装奶单号
- `milk_source_plan_order_number` - 奶源计划单号（关联外部系统）
- `license_plate_number` - 车牌号
- `status` - 状态（0:保存，1:审核）
- `first_milking_time` - 最早一滴挤奶时间
- `total_milk_capacity` - 装奶总量
- `net_weight` - 净重（过磅后）
- `is_push_milk_source` - 是否推送奶源系统

**关联表**：
- `op_milk_filling_order_detail` - 装奶单明细（奶仓分配）
- `op_milk_bin_quality_inspection` - 奶仓质检单

**业务说明**：
- 装奶单记录了奶车装载了哪些奶仓的奶，每个奶仓的装奶量
- 与奶样质检单（`op_milk_sample_quality_inspection`）关联，形成"装奶→质检→外运"的完整流程
- 推送到奶源系统后，工厂端可以查看送奶信息

### 2.4 牧场/饲料厂采样核心表（重点：双轨制逻辑）

#### 表10：`op_sampling_plan` - 原料取样计划主表（一卡通触发 OR 手动创建）

**作用**：原料（如玉米、豆粕等）进厂时的取样计划主表

**关键字段**：
- `sampling_plan_no` - 送检单号
- `supplier_code` / `supplier_name` - 供应商编码/名称
- `driver_name` / `driver_code` - 司机姓名/车牌号
- `car_in_time` / `car_out_time` - 进厂/出厂时间
- `status` - 状态（0未提交，1未取样，2已取样，3已作废）
- `sign_in_id` - 签到ID（一卡通系统的签到记录ID）
- `is_release` - 是否放行（0:未放行，1:已放行）

**状态流转**：
```
0 未提交 → 1 未取样 → 2 已取样
           ↓
        3 已作废
```

**触发方式（双模式）**：
1. **模式1（一卡通自动触发）**：一卡通系统司机签到后，调用接口 `/ranch/plan/notifySignIn`，自动创建 `op_sampling_plan` 记录
2. **模式2（手动创建）**：通过移动端"原料取样"页面人工新增

**关联Controller**：
- `OpSamplingPlanController.receiveSignInNotification()` - 接收一卡通签到通知
- `OpSamplingPlanController.add()` - 手动新增原料取样计划

---

#### 表11：`op_sampling_plan_sample` - 样品表（★核心！所有类型样品的统一存储）

**作用**：存储所有类型的样品信息（成品、库存、垫料、原料），是牧场/饲料厂质检的核心数据表

**关键字段**：
- `sampling_plan_id` - 取样计划主表ID（**注意**：如果是成品/垫料/库存，这里关联的是 `op_finished_product_sampling_plan_detail` 的ID；如果是原料，关联 `op_sampling_plan` 的ID）
- `sample_no` - 样品编号
- `invbill_code` / `invbill_name` - 物料编码/样品名称
- `sampling_type` - **样品类型**（0:成品，1:库存，2:垫料，3:原料）
- `new_sampling_types` - **新增取样类型**（1:牧场新增，2:饲料厂产品新增，3:饲料厂采购新增）
- `is_receive` - 是否接收（0:未接收，1:已接收）
- `whether_qualified` - 是否合格（A1:合格，R1:不合格）
- `is_push_sap` - 是否上传SAP（0:未上传，1:已上传）

**双轨制逻辑体现**：
- **路径A（PC端Excel导入）**：`sampling_plan_id` 关联 `op_finished_product_sampling_plan_detail` 的ID
- **路径B（移动端直接采样）**：`sampling_plan_id` 可能为空，直接插入样品记录（无计划主表）

**关联Controller**：
- `OpSamplingPlanController.addKCDL()` - 垫料/库存取样新增（移动端路径B）
- `OpFinishedProductSamplingPlanController.add()` - 成品取样计划新增（PC端路径A）

---

#### 表12：`op_sampling_plan_item` - 样品检测项目明细表

**作用**：记录每个样品需要检测的项目及其检测结果

**关键字段**：
- `sampling_plan_sample_id` - 关联样品表ID
- `item_id` / `item_code` / `item_name` - 检测项目ID/编码/名称
- `test_result` - 检测结果
- `qualitative_or_quantitative` - 定性/定量（1:定性，2:定量）
- `upper_limit` / `lower_limit` - 第一上限/下限
- `gf_upper_limit` / `gf_lower_limit` - 规范上限/下限

**说明**：每个样品可以有多个检测项目，化验员逐项录入检测结果

### 2.5 基础配置核心表

#### 表13：`bs_labtest_items` - 检测项目表（全局共享）

**作用**：所有业态共享的检测项目配置（如脂肪、蛋白、菌落总数等）

**关键字段**：

- `qualitative_or_quantitative` - 定性/定量（1:定性，2:定量）

#### 表14：`bs_invbill_item_standard` - 物料项目标准表

**作用**：物料的检验项目标准配置（不同物料、不同组织的标准可能不同）

**关键字段**：
- `invbill_code` - 物料编码
- `dept_id` - 组织ID
- `item_id` - 检测项目ID
- `upper_limit` / `lower_limit` - 第一上限/下限
- `gf_upper_limit` / `gf_lower_limit` - 规范上限/下限（更宽松的范围）
- `is_bj` - 是否必检（0:否，1:是）
- `is_cj` - 是否初检（0:否，1:是）

**判定逻辑**：
- 第一上下限：严格范围，超出为不合格
- 规范上下限：允许范围，超出第一限但在规范限内可能是"可接收"

### 2.7 软删除设计（系统级架构模式）

系统采用**软删除**（Soft Delete）架构，**不进行物理删除**，所有"删除"操作都是逻辑标记。

#### 2.7.1 软删除字段设计

| 字段名 | 数据类型 | 取值规则 | 说明 |
|--------|---------|----------|------|
| **delete_id** | VARCHAR | `'0'` 或 `当前记录的ID` | **高级设计**：解决 UNIQUE KEY 与软删除的冲突问题 |
| **is_delete** | VARCHAR | `'0'` 或 `'1'` | **简单设计**：常规软删除标记 |

#### 2.7.2 delete_id 的高级设计原理

**问题**：如果使用 `is_delete = '0'/'1'`，数据库唯一索引 `UNIQUE KEY (code, is_delete)` 会导致以下问题：
- 删除记录A后（`is_delete='1'`），再删除同名记录B，因为两条已删除记录的 `code` 相同且 `is_delete` 都是 `'1'`，违反唯一约束，插入失败

**解决方案**：`delete_id` 字段存储**当前记录的ID**而非简单的 `'0'/'1'`：
```sql
-- 未删除状态
delete_id = '0'

-- 删除时（赋值为记录自身的ID）
UPDATE op_test_result_base 
SET delete_id = id, update_time = NOW() 
WHERE id = 'xxx';

-- 结果：delete_id = 'a1b2c3d4-...'（记录的UUID）
```

**优势**：
- 每条已删除记录的 `delete_id` 都唯一（因为是各自的ID）
- 唯一索引 `UNIQUE KEY (code, delete_id)` 允许多条已删除的同名记录共存
- 未删除记录（`delete_id='0'`）仍然保证业务唯一性

**应用场景**：
- 允许删除后重新创建同名记录
- 保留完整的历史删除记录
- 适用于需要频繁"删除-重建"的业务场景（如检测项目作废后重新添加）

#### 2.7.3 使用 delete_id 的核心表

| 表名 | 软删除字段 | 说明 |
|------|-----------|------|
| `op_test_result_base` | `delete_id` | 化验单主表，作废时 `delete_id = id` |
| `op_test_result_info` | `delete_id` | 化验明细表，作废时 `delete_id = id` |
| `op_sampling_plan` | `delete_id` | 取样计划主表 |
| `op_sampling_plan_sample` | `is_delete` | 样品表，使用简单软删除 `'0'/'1'` |
| `op_sample_sap_batch` | `is_delete` | SAP批次绑定表 |

#### 2.7.4 软删除的业务语义

| 操作 | 软删除标记 | 业务含义 | SQL示例 |
|------|-----------|----------|---------|
| **作废** | `delete_id = id` 或 `is_delete='1'` | 记录失效，不可再使用，但保留历史 | 化验单作废、样品作废 |
| **归档** | `retest_flag='1'` | 复检场景下，原记录归档，不再显示在待化验列表 | 复检审核通过后 |
| **物理删除** | 直接 DELETE | 仅用于测试数据清理，**生产环境禁止** | - |
