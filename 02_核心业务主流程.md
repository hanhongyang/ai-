## 三、整体业务理解（上帝视角）

### 3.1 系统定位

**光明LIMS质检系统** 是一套面向**乳制品行业**的全流程质量检验管理系统，覆盖从**奶源采集 → 样品检测 → 结果判定 → 报告生成**的完整链路。

**适用场景**：
- 牧场奶源质量检验
- 饲料原料与成品检验
- 中心实验室化验检测
- 工厂出厂质量检验

### 3.2 核心业务角色（推测）

| 角色类型 | 主要职责 | 使用端 |
|---------|---------|--------|
| **采样员** | 现场采集样品、录入采样信息、生成铅封号 | 移动端（钉钉小程序） |
| **接样员** | 接收样品、样品登记、分配化验任务 | 移动端 + PC端 |
| **化验员** | 执行检测、录入检测结果、仪器操作 | PC端（管理后台） |
| **审核员/质检主管** | 审核检测结果、判定合格/不合格、签发报告 | PC端 |
| **实验室管理员** | 管理仪器设备、试剂耗材、人员培训 | PC端 |
| **数据分析员** | 生成统计报表、质量趋势分析 | PC端 + 移动端 |
| **系统管理员** | 配置检验标准、管理权限、对接SAP | PC端 |

### 3.3 核心业务对象

1. **委托单（Entrust Order）**：送检单位提交的检测申请，包含样品信息和检测项目
2. **样品（Sample）**：需要检测的物料（奶样、饲料、血样等）
3. **检验项目（Lab Test Item）**：需要检测的指标（如蛋白质、脂肪、体细胞数等）
4. **检验方法（Lab Test Method）**：检测使用的方法和仪器
5. **检验标准（Standard）**：合格判定的阈值范围
6. **检测结果（Test Result）**：化验员录入的实际检测数据
7. **检验报告（Report）**：汇总的质检报告文档

### 3.4 系统集成关系

```
钉钉小程序（移动端）
    ↓
LIMS系统（核心）
    ↓
SAP系统（ERP）- 物料主数据、客户信息、质检结果回传
    ↓
光明一卡通（gmykt）- 人员认证
```

### 3.2 原料取样流程（区分一卡通接口触发 VS 手动创建）

#### 3.2.1 流程总览

原料取样有**两种触发模式**，数据最终都流向 `op_sampling_plan` + `op_sampling_plan_sample`：

```
【模式1：一卡通自动触发】
一卡通系统司机签到 
   ↓
调用接口 /ranch/plan/notifySignIn
   ↓
自动创建 op_sampling_plan 记录（status=1 未取样）
   ↓
采样员扫码或手动关联，生成 op_sampling_plan_sample
   ↓
化验录入 → 判定 → 放行

【模式2：手动创建】
移动端"原料取样"页面
   ↓
手动创建 op_sampling_plan 记录
   ↓
生成 op_sampling_plan_sample
   ↓
化验录入 → 判定
```

#### 3.2.2 模式1：一卡通接口自动触发（深度代码审计）

**接口路径**：`/ranch/plan/notifySignIn`

**Controller方法**：`OpSamplingPlanController.receiveSignInNotification()`

**请求DTO**：`SignInNotificationDTO`

**关键字段**：
- `id` - 签到主键ID（一卡通系统的签到记录ID）
- `driverName` / `driverPhone` / `driverCode` - 司机信息/车牌号
- `signInTime` - 签到时间
- `destinationCode` - 目的地编码（LIMS的牧场编码）
- `fileBase64` / `originalFilename` - 随车检验报告（Base64附件）

**Service逻辑**：`OpSamplingPlanServiceImpl.processSignInNotification()`

**处理步骤**：
1. 接收一卡通系统的签到通知（JSON数据 + Base64附件）
2. 根据 `destinationCode` 查询LIMS系统的 `dept_id`
3. 创建 `OpSamplingPlan` 对象：
   - `sign_in_id` = notifyDTO.id（关联一卡通签到记录）
   - `driver_name` / `driver_code` = 司机信息
   - `car_in_time` = signInTime
   - `status` = '1'（未取样）
   - `dept_id` = 查询到的LIMS部门ID
4. 生成 `sampling_plan_no`（送检单号）
5. 处理Base64附件，保存到服务器，生成 `car_file_id`
6. 插入 `op_sampling_plan` 表

**插入的表**：`op_sampling_plan`

**初始状态**：`status = '1'`（未取样）

**后续流程**：
- 采样员在移动端扫码或选择该计划，添加样品到 `op_sampling_plan_sample` 表
- `sampling_plan_id` 关联 `op_sampling_plan.op_sampling_plan_id`
- `sampling_type = '3'`（原料）

### 3.3 成品/库存/垫料取样流程（明确双轨制逻辑）

#### 3.3.1 流程总览：PC端Excel导入 VS 移动端直接采样

成品/库存/垫料取样存在**两条平行路径**，数据模型完全不同：

```
【路径A：PC端批量导入（计划-明细模式）】
PC端《取样计划》页面 
   ↓
上传Excel文件
   ↓
插入 op_finished_product_sampling_plan（主表）
   ↓
插入 op_finished_product_sampling_plan_detail（明细表）
   ↓
移动端扫码或选择计划，添加样品到 op_sampling_plan_sample
   ↓
化验录入 → 判定 → 上传SAP

【路径B：移动端手动新增（无计划主表）】
钉钉小程序直接操作
   ↓
直接插入 op_sampling_plan_sample（★无主表计划）
   ↓
化验录入 → 判定 → 上传SAP
```

**关键区别**：
- **路径A**：有 `op_finished_product_sampling_plan` 主表，`op_sampling_plan_sample.sampling_plan_id` 关联明细表ID
- **路径B**：**无**主表计划，`op_sampling_plan_sample.sampling_plan_id` 可能为空或关联其他逻辑

---

#### 3.3.2 路径A：PC端Excel批量导入（计划-明细模式）

**Controller方法**：`OpFinishedProductSamplingPlanController.importData()`

**涉及表**：
- `op_finished_product_sampling_plan` - 采样计划主表
- `op_finished_product_sampling_plan_detail` - 采样计划明细表
- `op_sampling_plan_sample` - 样品表（后续关联）

**业务流程**：

**Step 1：Excel导入创建计划**
1. 用户在PC端下载Excel模板
2. 填写物料信息：
   - `material_code` / `material_name` - 物料编码/名称
   - `production_order_number` - 生产订单号
   - `planned_sample_quantity` - 计划取样份数
3. 上传Excel，系统解析后：
   - 插入 `op_finished_product_sampling_plan`（主表）
     - `plan_type` = 0:成品/1:库存/2:垫料
     - `status` = '1'（未取样）
     - `dept_id` = 当前用户的部门ID
   - 插入 `op_finished_product_sampling_plan_detail`（明细表）
     - 每行Excel数据生成一条明细记录

**Step 2：移动端扫码或选择计划**
1. 采样员在钉钉小程序中查看待采样的计划
2. 扫描二维码或手动选择计划明细
3. 添加样品信息后，插入 `op_sampling_plan_sample`：
   - `sampling_plan_id` = `op_finished_product_sampling_plan_detail.finished_product_sampling_plan_detail_id`（★关联明细表ID）
   - `sampling_type` = 0:成品/1:库存/2:垫料
   - `sample_no` - 样品编号（自动生成或手动录入）

**Step 3：化验录入**
- 化验员在PC端或移动端录入检测结果
- 更新 `op_sampling_plan_item` 表
- 判定合格/不合格，更新 `op_sampling_plan_sample.whether_qualified`

**Step 4：更新计划状态**
- 所有样品采样完成后，更新 `op_finished_product_sampling_plan.status = '2'`（已取样）

##### 【Step 1】开始化验 - 批量转化待检任务为化验单

**接口**：`POST /ranch/testResult/startTest`
**方法**：`OpTestResultBaseServiceImpl.startTest(List<String> opSamplingPlanItemIds)`（line 763-887）

**业务逻辑**：
1. **前置校验**：
   - 所有选中的 `op_sampling_plan_item` 必须属于**相同的检测项目**（`item_id` 一致）
   - 使用 **Redisson 分布式锁** 防止重复点击（lockKey: `assay:start:lock:{itemIds}`）
   - 双重检查：获取锁后立即查询数据库，防止并发创建重复记录（`countExistingByPlanItemIds`）
   
2. **核心操作**（多表联合事务）：
   - **INSERT** `op_test_result_base`（化验单主表）：
     - `id` = UUID
     - `result_no` = 自动生成化验单号（`codeGeneratorUtil.generateHydAssayOrderNo(deptId)`）
     - `item_name` / `item_id` = 检测项目名称/ID（所有待检任务共享）
     - `test_user` / `test_user_id` = 当前用户
     - `test_time` = 当前时间
     - `status` = '2'（待提交）★ 注意：不是 '1'，直接跳过待化验状态
   - **批量 INSERT** `op_test_result_info`（化验明细表）：
     - 每个 `op_sampling_plan_item` 生成一条 `op_test_result_info` 记录
     - `base_id` = 上面创建的 `op_test_result_base.id`
     - `plan_item_id` = `op_sampling_plan_item.op_sampling_plan_item_id`（关联原始待检任务）
     - `plan_sample_id` = 样品ID
     - `sample_no` / `invbill_name` = 从 `op_sampling_plan_sample` 表 JOIN 查询
     - `feature_id` / `upper_limit` / `lower_limit` = 复制检测特性和限值
     - `status` = '2'（待提交）
     - `is_push_sap` = '0'（未推送SAP）
     - `retest_flag` = '0'（非复检记录）
   
3. **返回值**：新创建的 `op_test_result_base.id`（前端用于跳转到化验单页面）

**状态流转**：
```
op_sampling_plan_item（待化验任务）
   ↓ startTest()
op_test_result_base（status='2' 待提交）+ op_test_result_info（status='2'）
```

##### 【Step 3】审核通过 - 生成不合格处理单

**接口**：`PUT /ranch/testResult/approve/{id}`（PC端）或 `POST /ranch/testResult/approveTest`（手机端）
**方法**：`OpTestResultBaseServiceImpl.approveTestResult(String id)`（line 974-1008）

**业务逻辑**：
1. 更新 `op_test_result_base`：
   - `status` = '4'（已审核）
   - `examine_time` / `examine_user_id` / `examine_user` = 审核人和时间
   - `return_reason` = null（清空退回原因）
2. 批量更新 `op_test_result_info`：
   - `status` = '4'（已审核）
3. **化验不合格处理**（关键联动）：
   - 遍历所有 `op_test_result_info`，如果 `check_result = '2'`（不合格）
   - 调用 `handleUnquality(info)` 方法（line 1010-1068）：
     - 查询关联的 `op_sampling_plan`（获取签到ID、司机信息）
     - 查询关联的 `op_sampling_plan_sample`（获取供应商信息）
     - **INSERT** `op_sample_unquality`（不合格品主表）：
       - `source_id` = `plan_sample_id`（样品ID）
       - `sign_in_id` = 签到ID（如果有）
       - `material_code` / `material_name` = 物料编码/名称
       - `ctype` = '化验不合格'
       - `processing_type` = '1'
       - `status` = '0'（待处理）
     - **INSERT** `op_sample_unquality_detail`（不合格品明细表）：
       - `item_name` / `feature_name` / `result` / `check_result` = 检测项目和结果
   - ★ 自动触发不合格品管理流程

**状态流转**：
```
status='3'（待审核）→ status='4'（已审核）
   ↓（如果 check_result='2'）
INSERT op_sample_unquality（不合格品处理单）
```

2. 调用 `approveRetestSingle(id)`（★关键逻辑审计发现）：
   
   **旧版本逻辑**（`approveRetestSingleold` line 365-452，已弃用但代码保留）：
   - 将原 `op_test_result_info` 的 `retest_flag` 改为 '1'（已复检/归档）
   - **复制原记录并 INSERT 新的 `op_test_result_info`**：
     - `id` = 新UUID
     - `result` / `check_result` = null（清空化验结果）
     - `retest_flag` = '0'（正常待化验记录）
     - 其他字段完全复制原记录
   - ★ 结论：**复检是INSERT新记录，不是UPDATE覆盖原记录**
   
   **新版本逻辑**（line 334-361，当前使用）：
   - 只将 `retest_flag` 改为 '1'（已失效/归档）
   - ★ **不 INSERT 新记录**，而是依赖待化验列表的 SQL 逻辑自动显示

**Phase C：化验员在待化验列表看到复检项目**

**核心机制**：`OpTestResultBaseMapper.xml.selectPendingTaskList`（line 381-490）

**关键 SQL 逻辑**：
```sql
LEFT JOIN op_test_result_info tri
ON pi.op_sampling_plan_item_id = tri.plan_item_id
AND tri.delete_id = '0'
-- ★ 关键：只关联 retest_flag='0' 或 '2' 的记录
AND (tri.retest_flag = '0' OR tri.retest_flag = '2' OR tri.retest_flag IS NULL)

WHERE
...
-- 如果原记录的 retest_flag='1'，则不满足上面的 JOIN 条件
-- tri.id 会是 NULL，满足待化验条件，重新显示在列表中
AND (tri.id IS NULL OR otrb.status = '1')
```

**工作原理**：
1. 复检审核通过后，原 `op_test_result_info` 的 `retest_flag` 被改为 '1'
2. 待化验列表查询时，LEFT JOIN 条件 `retest_flag='0' OR '2'` **不匹配** `retest_flag='1'` 的记录
3. JOIN 失败导致 `tri.id IS NULL`，该 `op_sampling_plan_item` 重新满足"待化验"条件
4. 化验员点击"开始化验"时，调用 `startTest()`，会 **INSERT 新的 `op_test_result_info`**（`retest_flag='0'`）
5. 新记录与原记录并存，原记录作为历史归档（`retest_flag='1'`）

**★ 设计精髓**：
- 不需要在审核环节 INSERT 新记录
- 利用 SQL JOIN 条件自动过滤归档记录
- 由化验员"开始化验"操作触发真正的新记录创建
- 保留完整的复检历史链（原记录不删除，只标记为 '1'）

**复检相关SQL**：
- `OpTestResultInfoMapper.xml.updateRetestFlagById`（line 456-462）：简单的 UPDATE retest_flag
- `OpTestResultInfoMapper.xml.selectRetestDetail`（line 488-493）：查询 `retest_flag='2'` 的待审核复检记录
- `OpTestResultInfoMapper.xml.countExistingByPlanItemIds`（line 527-536）：检查重复时排除 `retest_flag != '0'` 的记录

**retest_flag 状态机**：
```
'0' = 正常记录（非复检）
'1' = 已复检/归档（批量复检时）或 已失效（单条复检审核通过后）
'2' = 复检待审核（单条复检申请时）
```

**C. 推送化验结果到SAP**

**方法**：`OpTestResultBaseServiceImpl.pushSap(OpSamplingPlanSamplePushSapDTO opSamplingPlanSample)`（line 466-592）

**核心逻辑**（复杂的分批推送 + 状态控制）：

1. **预判化验是否全部完成**（line 486-493）：
   - 查询该样品的所有 `op_test_result_info`
   - 如果有 `result` 和 `check_result` 都为空的记录，说明还有未完成的化验项
   - 设置标志位 `isAllFinished = true/false`

2. **循环推送每个批次**（line 513-568）：
   - 遍历 `op_sample_sap_batch` 列表（一个样品可能关联多个批次）
   - 跳过 `upload_status='2'` 的批次（已推送完成）
   - 组装 SAP 接口参数（`CheckResRetJudgeDTO`）：
     - `WERK` = 工厂编码（从 `sys_dept.sap_name` 查询）
     - `PRUEFLOS` = 检验批次
     - `VCODE` = 合格判定（A1/R1）
     - `ZTYPE` = 推送类型：
       - **'H'** = 首次推送（`upload_status='0'`）
       - **'I'** = 增量推送（`upload_status='1'`，中途上报过）
     - `ITEMS` = 检测项目列表（只推送已完成的项目）：
       - `VERWMERKM` = 项目名称（`item_name`）
       - `ORIGINAL_INPUT` = 检测结果（`result`）
   - 调用 SAP 接口 `sapUtils.checkResRetJudge()`
   - **根据推送成功 + 化验完成度，更新批次状态**（line 554-557）：
     - 如果 `isAllFinished=true`（所有化验项完成）→ `upload_status='2'`（推送完成）
     - 如果 `isAllFinished=false`（还有未完成项）→ `upload_status='1'`（部分推送）

3. **宏观控制：统一更新主表和明细表状态**（line 571-588）：
   - **条件**：本次有成功记录 且 没有失败批次（`successCount > 0 && allBatchesSuccess`）
   - 更新 `op_sampling_plan_sample.is_push_sap`：
     - 如果 `isAllFinished=true` → `is_push_sap='2'`（推送完成）
     - 如果 `isAllFinished=false` → `is_push_sap='1'`（部分推送）
   - 批量更新 `op_test_result_info.is_push_sap = '1'`

**is_push_sap / upload_status 状态机**：
```
'0' = 未推送（初始状态）
'1' = 部分推送（有未完成的化验项，已推送一次）
'2' = 推送完成（所有化验项完成，已推送最后一次）
```

**推送时机**：
- 可以在化验未完成时推送（部分结果）
- 化验完成后再次推送（全部结果）
- SAP系统根据 `ZTYPE='H'/'I'` 区分首次和增量

### 3.5 饲料化验单状态3后的被动等待机制

**业务场景**：饲料化验单完成校对后的流转逻辑

当饲料化验单达到状态3（校对完成 JDWC）后，系统**不会主动触发任何后续接口或推送动作**。化验单进入**被动等待状态**，作为数据源等待其他模块查询。

**状态3的特征**：
- ✅ 检测人员已录入完成并提交
- ✅ 校对人员已核对数据并签字
- ❌ **不会**自动触发审核流程（状态4已废弃）
- ❌ **不会**主动推送数据到报告模块
- ✅ 作为**终态**存在于 `op_jczx_feed_result_base` 表中

**后续流转机制**：
```
饲料化验单 (status=3, 校对完成)
   ↓
【被动等待】
   ↓
饲料报告模块主动查询
   - 查询条件：WHERE status='3' AND item_id=? AND entrust_order_id=?
   - 读取化验数据生成报告
   ↓
报告生成完成后：
   - 报告表记录委托单ID和化验单ID的关联关系
   - 化验单状态保持为3（不变）
   - 报告状态独立流转（待审核 → 已审核 → 已发送）
```

**技术说明**：
- 饲料化验单与报告模块采用**松耦合设计**
- 化验单只负责数据生产，报告模块负责数据消费
- 查询权限由报告模块控制，化验单无需感知报告生成时机

## 五、中心化验室业务流程（Central Laboratory）

**⚠️ 关键认知**：中心化验室与牧场/饲料厂的最大区别是**委托制**！其他厂区或供应链部门送样品到中心化验室，由中心化验室接收、化验、出具报告。

### 5.1 业务概述

**核心定位**：中心化验室是集团的**检测服务中心**，接收来自牧场、饲料厂、外部供应商的委托检验申请。

**核心表结构**：
- `op_blood_entrust_order` + `op_blood_entrust_order_sample` + `op_blood_entrust_order_item` - 血样委托单
- `op_pcr_entrust_order` + `op_pcr_entrust_order_sample` + `op_pcr_entrust_order_item` - PCR委托单
- `op_feed_entrust_order` + `op_feed_entrust_order_sample` + `op_feed_entrust_order_item` - 饲料委托单
- `op_sample_receive` - 样品接收记录
- `op_jczx_*_result_base` / `op_jczx_*_result_info` - 化验结果表（血样、PCR、饲料各有独立表）
- `op_jczx_*_report_base` / `op_jczx_*_report_info` - 报告表

**后端包**：`com.gmlimsqi.business.labtest`

**前端位置**：`lims-vue/src/views/CentralLaboratory`（仅PC端，移动端无法填写委托单）

### 5.2 三种委托单对比

### 5.7 变更日志机制（OpBloodEntrustOrderChangeLog）

**业务背景**：委托方修改委托单时，检测中心需要知道改了什么，以便重新评估检测方案。

**触发时机**：`OpBloodEntrustOrderServiceImpl.updateOpBloodEntrustOrder()` line 418-562

**记录范围**：
- **主表字段**：检测项目、送检单位、送检时间、地址、送样人
- **样品字段**：管号、牛号、类别、备注、配种天数、母牛号

**日志表结构** (`op_blood_entrust_order_change_log`):
- `log_id` - 日志主键
- `business_id` - 关联ID（委托单ID 或 样品ID）
- `business_type` - 业务类型（ORDER / SAMPLE）
- `field_key` - 字段键（如 sampleName）
- `field_name` - 字段名称（如 牛号）
- `old_value` - 修改前的值
- `new_value` - 修改后的值
- `create_by` - 修改人（用户名）

**前端展示**：委托单详情页会显示"变更日志"列表，检测中心可查看委托方的所有修改记录

**辅助方法**：`compareAndLog()` line 610-627
```java
private void compareAndLog(String businessId, String type, String fieldKey, String fieldName,
                           String oldVal, String newVal, List<OpBloodEntrustOrderChangeLog> list, String user) {
    String o = oldVal == null ? "" : oldVal.trim();
    String n = newVal == null ? "" : newVal.trim();
    if (!o.equals(n)) {
        // 创建日志记录
        OpBloodEntrustOrderChangeLog log = new OpBloodEntrustOrderChangeLog();
        log.setLogId(IdUtils.simpleUUID());
        log.setBusinessId(businessId);
        log.setBusinessType(type);
        log.setFieldKey(fieldKey);
        log.setFieldName(fieldName);
        log.setOldValue(o);
        log.setNewValue(n);
        log.setCreateBy(user);
        list.add(log);
    }
}
```

### 5.9 饲料化验动态路由机制（核心技术底牌）

#### 5.9.1 业务背景

#### 5.9.2 路由判断核心字段：`modelNo` (模板编号)

**完整路由映射表** (`feedTask/index.vue` lines 1186-1213)：

### 6.2 数据拉取机制：如何识别"可制作报告"的化验单？

#### 6.2.1 业务规则（精确描述）

**❌ 常见误解**：报告模块查询 `op_jczx_feed_result_base.status='3'` 的化验单

**✅ 真实逻辑**：报告模块检查 `op_jczx_feed_result_info.check_user IS NOT NULL`

#### 6.2.2 核心 SQL 逻辑（待制作列表）

**Mapper方法**：`OpJczxFeedReportBaseMapper.selectOpJczxFeedReportBaseListStatus0`

**关键WHERE条件**：

```sql
-- 1. 样品报告表中不存在（排除已制作的报告）
AND s.sample_no NOT IN (
    SELECT b.sample_no
    FROM op_jczx_feed_report_base b
    WHERE b.sample_no IS NOT NULL 
    AND b.delete_id='0' 
    AND b.status!='6'
)

-- 2. 必须有化验数据记录
AND EXISTS (
    SELECT 1
    FROM op_jczx_feed_result_info i
    WHERE i.entrust_order_sample_id = s.op_feed_entrust_order_sample_id 
    AND i.delete_id='0' 
    AND i.is_reset='0'
)

-- 3. 所有检测项目都已完成校对（核心条件！）
AND NOT EXISTS (
    SELECT 1
    FROM op_feed_entrust_order_item oi
    LEFT JOIN op_jczx_feed_result_info i
        ON i.entrust_order_sample_id = oi.op_feed_entrust_order_sample_id
        AND oi.op_feed_entrust_order_item_id = i.entrust_order_item_id
        AND i.delete_id = '0'
        AND i.is_reset = '0'
    WHERE oi.op_feed_entrust_order_sample_id = s.op_feed_entrust_order_sample_id
    AND oi.is_delete = '0'
    -- 【关键判断】：只要有任何一个项目未录入或未校对，就排除整个样品
    AND (i.op_jczx_feed_result_info_id IS NULL OR i.check_user IS NULL)
)
```

**业务大白话解释**：
- `check_user` 字段是在化验单状态变为 **3（校对完成）** 时被填充的
- 只有当样品下**所有检测项目**都有 `check_user`（即都完成校对）时，该样品才会出现在"待制作"列表
- 这就是"被动等待"的实现机制

#### 6.10.3 饲料报告的"状态跳跃"实现机制

**问题核心**：饲料报告前端Tab中status=1被注释，用户看不到"制作中"状态，那数据如何从status=0跳到status=2？

##### 关键代码路径

**列表页跳转逻辑** (`feed/index.vue:1231-1247`)：


**详情页按钮渲染逻辑** (`reportgensl.vue:396-403`)：


```javascript

        }
```

##### 状态跳跃时序图

```
用户操作                    前端代码                      后端状态                   Tab显示
─────────────────────────────────────────────────────────────────────────────────────
```

**关键结论**：
1. 用户从Tab 0点击"编制"，进入详情页
2. 用户点击"保存"按钮，前端**连续调用两次API**
3. 第一次调用：`status='1'`（制作中）
4. 第二次调用：`status='2'`（待审核）
5. 用户视角：点一次"保存"，报告直接出现在"待审核"Tab中
6. **status=1在数据库中存在的时长 < 1秒**（仅作为两次API调用之间的瞬时中间态）

#### 6.10.5 技术债务影响分析

##### ✅ 优点

1. **用户体验简化**：隐藏不必要的中间状态，减少操作步骤
2. **向后兼容**：后端保留完整状态机，未来可随时恢复被隐藏的Tab
3. **数据库完整性**：所有状态流转记录完整，便于审计追溯

##### ⚠️ 缺点

1. **代码可读性差**：`pdfbc()`方法名为"保存"，实际却执行"保存+提交"双重操作
2. **调试难度高**：前端注释Tab导致开发者误认为status=1不存在
3. **状态不一致风险**：如果第一次API成功、第二次API失败，报告会卡在status=1（无Tab显示，成为"幽灵状态"）
4. **文档缺失**：历史需求变更未记录，新人接手难以理解设计意图

##### 🔧 潜在风险

**场景**：网络波动导致两次API调用不连续

```javascript
pdfbc() {
    Reportslbc({ status: '1' }).then(res => { // ✅ 成功，status=1写入数据库
        this.pdfMtj(); // ❌ 网络中断，未执行
    });
}
```

**结果**：
- 数据库状态：status='1'
- 前端Tab：无任何列表显示该报告（Tab 1被注释）
- 用户视角：报告消失了（实际在数据库中，但前端无入口访问）

**临时修复方案**：
1. 管理员直接修改数据库：`UPDATE op_jczx_feed_report_base SET status='2' WHERE status='1'`
2. 或在后端Controller新增兜底接口，定时扫描status=1超过5分钟的记录自动提交
