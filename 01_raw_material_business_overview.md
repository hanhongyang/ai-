## 1. 模块定位

### 这个模块解决什么业务问题

牧场和饲料厂每天都会接收大量饲料原料（如玉米、豆粕、棉籽等）。这些原料的质量直接影响奶牛健康和乳品品质。原料取样化验模块解决的核心问题是：

**每一车运进牧场的原料，在入库使用之前，都必须经过"取样 → 化验 → 判定 → 放行"这一整套质量检验流程，确保不合格原料不会进入生产环节。**

### 五个关键环节之间的关系

```
取样 → 化验 → 判定 → 放行
 ↑       ↑      ↑       ↑
采样员  化验员  系统+审核  放行人员/系统
```

- **取样**：采样员从运料车上物理采集原料样品，记录样品信息。没有取样，后面所有环节都无法进行。
- **化验**：化验员对样品进行检测（如水分、蛋白、毒素等指标），录入检测结果。这是质量判断的依据。
- **判定**：系统根据预设的质量标准，自动比对化验结果，判定每个检测项目是"合格"还是"不合格"。审核员确认判定结果。
- **放行**：当一年原料的所有检测项目全部合格后，系统标记该车原料"已放行"，允许入库使用。一卡通系统可以查询这个放行状态。

### 四个角色在流程中的作用

| 角色 | 在流程中的位置 | 做什么 | 对流程的影响 |
|------|-------------|--------|-------------|
| **司机** | 流程起点 | 在一卡通系统签到，触发取样计划生成 | 不签到→系统不会自动创建取样计划（但可通过手动方式补建） |
| **采样员** | 取样环节 | 到现场从车上采集原料样品，在移动端录入样品信息 | 不取样→没有样品→无法化验→无法放行 |
| **化验员** | 化验环节 | 按照检测标准对样品进行各项指标检测，在系统中录入结果 | 不化验→没有结果→无法判定→无法放行 |
| **审核员** | 判定环节 | 确认化验结果的准确性，最终确认合格或不合格 | 不审核→化验结果未生效→放行状态不会更新 |

## 2. 适用业务范围

### 明确适用的场景

- **牧场原料进厂质检**：牧场采购的饲料原料（玉米、豆粕、棉籽等）运到牧场时，需要进行质量检验。
- **饲料厂原料进厂质检**：饲料厂采购的原料进厂时，同样需要进行质量检验。

### 关键业务标识

在系统中，原料取样业务的数据通过以下方式标识：

- **样品类型（sampling_type）= 3**：表示这是一条"原料"样品记录，区别于成品（0）、库存（1）、垫料（2）。
- **数据存储在原料取样计划表（op_sampling_plan）**：每一车原料的取样任务都对应一条取样计划记录。
- **样品数据存储在样品表（op_sampling_plan_sample）**：每个从原料上采集的样品都在这里记录。
- **检测项目存储在检测项目表（op_sampling_plan_item）**：每个样品需要检测哪些指标、检测结果是什么，都在这里记录。

## 3. 不适用范围

以下业务场景**不属于**本模块，请不要用本模块的知识来回答相关问题：

| 不属于本模块的业务 | 属于哪个模块 | 典型特征 |
|-------------------|------------|---------|
| **中心化验室委托单流程** | 中心化验室（labtest） | 外部单位送样到检测中心，走"委托单→受理→检测→报告"流程 |
| **奶源质检流程** | 牧场奶源质检 | 检测对象是生鲜乳（牛奶），不是饲料原料；使用独立的奶样质检表 |
| **成品取样** | 饲料厂成品检验 | 检测对象是饲料成品（生产出来的饲料），不是进货原料 |
| **库存取样** | 库存物料抽检 | 检测对象是仓库里存放的物料，不是新进厂的原料 |
| **垫料取样** | 牧场垫料检验 | 检测对象是牛床垫料，不是饲料原料 |
| **饲料报告生成** | 中心化验室报告模块 | 属于中心化验室委托单的后续环节，生成正式检测报告 |
| **SAP 复杂推送** | 系统集成模块 | 化验结果推送到 SAP 系统，属于数据集成层面 |
| **自动审核** | 当前系统不支持 | 当前 MVP 阶段审核需要人工确认 |
| **自动放行** | 当前系统不支持 | 放行操作需要相关人员确认 |
| **实时状态查询** | 当前 AI 不支持 | AI 不能直接查询数据库获取某辆车的当前状态 |

## 4. 原料取样的两种触发方式

### 4.1 一卡通签到自动触发

**业务场景**：

司机驾驶运料车到达牧场/饲料厂门口，在一卡通系统上签到（类似于"报到"）。签到成功后，一卡通系统会自动通知 LIMS 质检系统，LIMS 系统为这车原料自动创建一条原料取样计划。

**具体过程**：

1. 司机在一卡通系统完成签到。
2. 一卡通系统调用 LIMS 的接口（`/ranch/plan/notifySignIn`），把签到信息发送给 LIMS。
3. LIMS 收到通知后，自动创建一条原料取样计划记录：
   - 记录司机的姓名和车牌号。
   - 记录签到时间（作为进厂时间）。
   - 记录一卡通的签到 ID（`sign_in_id`），用于后续两个系统之间的关联。
   - 原料取样计划的初始状态为"未取样"（status=1）。
4. 如果司机随车带有检验报告（纸质或电子），一卡通系统会把报告文件一并传给 LIMS 保存。

**需要注意**：

- 签到只是创建了取样计划，**不等于已经取样**。
- 采样员仍然需要根据这条取样计划去现场完成实际的样品采集。
- 如果一卡通系统故障或司机未签到，仍然可以通过方式二（手动创建）来补建取样计划。

### 4.2 移动端手动创建

**业务场景**：

采样员或其他工作人员在钉钉小程序上，手动填写原料取样信息，创建取样计划。

**适用情况**：

- 一卡通系统未触发自动创建（如签到失败、系统故障）。
- 临时或补录的原料取样需求。

**具体过程**：

1. 工作人员打开钉钉小程序中的原料取样页面：
   - 饲料厂人员使用：`饲料厂检验 → 原料取样` 页面。
   - 牧场人员使用：`牧场检验` 中的原料取样功能。
2. 手动填写以下信息：
   - 供应商编码和名称（原料是哪个供应商提供的）。
   - 司机姓名和车牌号（哪辆车运来的）。
   - 进厂时间（车辆什么时候到的）。
3. 提交后，系统创建一条原料取样计划记录，初始状态同样为"未取样"（status=1）。
4. 后续的取样、化验、判定、放行流程与方式一完全相同。

**两种方式的对比**：

| 对比维度 | 一卡通签到触发 | 移动端手动创建 |
|---------|-------------|-------------|
| 创建者 | 一卡通系统自动触发 | 人工在钉钉小程序中创建 |
| 是否有签到ID | 有（sign_in_id） | 无 |
| 一卡通能否查询放行状态 | 能 | 可能不能（无签到ID关联）【待确认】 |
| 适用场景 | 正常签到进厂 | 补录、故障恢复、临时需求 |
| 后续流程 | 完全相同 | 完全相同 |

## 5. 主流程说明

以下用"一车玉米运到牧场"为例，说明完整的业务流程。

### 第一步：生成原料取样计划

司机驾驶运料车到达牧场，在一卡通系统签到。系统自动为这车原料创建了一条"原料取样计划"，状态为"未取样"。这条计划相当于一个"待办任务"，告诉采样员：有一车新原料到了，需要去取样。

如果一卡通没有自动生成，采样员也可以在钉钉上手动创建这条计划。

### 第二步：采样员完成取样

采样员在钉钉小程序上看到"未取样"的计划，前往现场从运料车上采集原料样品。采集完成后，采样员在系统中录入样品的基本信息（物料名称、批次等），系统为这个样品生成一个样品编号。

此时，原料取样计划的状态从"未取样"变为"已取样"，表示现场采样工作已经完成。

### 第三步：系统生成样品记录

系统为采集的样品在样品表中创建一条正式记录，标记样品类型为"原料"（sampling_type=3）。这条记录后续会关联化验数据和判定结果。

### 第四步：系统生成或关联检测项目

系统根据该原料的物料编码和所在牧场/饲料厂的标准配置，自动确定这个样品需要检测哪些项目。例如：

- 玉米可能需要检测：水分、粗蛋白、黄曲霉毒素等。
- 豆粕可能需要检测：粗蛋白、粗纤维等。

每个检测项目都会生成一条待化验任务，等待化验员处理。

**【待确认】**：检测项目自动生成的具体规则——是按物料编码匹配所有"必检"项目，还是有其他筛选逻辑（如根据供应商、季节等条件动态调整）。

### 第五步：化验员录入检测结果

化验员在 PC 端管理后台看到待化验的任务。化验员勾选属于同一检测项目的样品，点击"开始化验"，系统生成一张化验单。

化验员按照检测标准，在实验室完成检测后，在化验单中逐项录入检测结果（如水分含量 14.5%、粗蛋白含量 8.2% 等）。录入完成后点击"提交"，化验单状态变为"待审核"。

### 第六步：系统根据标准判定合格或不合格

审核员（通常是质检主管）在系统中查看化验结果，确认数据无误后点击"审核通过"。

审核通过时，系统会自动将每项检测结果与预设的质量标准（如玉米水分不得超过 15%）进行比对：

- 检测结果在标准范围内 → 判定为"合格"。
- 检测结果超出标准范围 → 判定为"不合格"。

如果有任何检测项目不合格，系统会自动生成一条"不合格品处理记录"，标记该原料需要进一步处理。

### 第七步：全部合格后允许放行

当一车原料的所有检测项目都判定为合格后，系统将该车原料标记为"已放行"（is_release = 1），表示这车原料可以入库使用了。

**重要说明**：放行是最终确认动作。只要还有一个样品的结果未出、或有一个检测项目被判为不合格，这车原料就不能放行。

### 第八步：一卡通系统可以查询放行状态

一卡通系统可以通过接口查询某次签到对应的原料是否已经放行。司机或相关人员在需要确认放行状态时，可以借助一卡通系统的查询功能获取结果。

## 6. 核心业务对象说明

### 6.1 原料取样计划

**用户视角含义**：
每来一车新原料，系统就会生成一份"取样计划"，相当于一个"取样任务单"。它记录了这车原料的基本信息——是谁送来的、哪辆车运的、什么时候到的，以及当前处理到了什么阶段。

一份取样计划代表"一车原料的一次进厂质检任务"。

**系统中对应的数据表**：`op_sampling_plan`

**这份计划有几个关键状态**：
- **未提交（0）**：草稿，还没正式生效。
- **未取样（1）**：计划已生效，采样员需要去取样。这是签到后或手动创建后的初始状态。
- **已取样（2）**：采样员已经完成了现场采样。
- **已作废（3）**：这个计划被取消了，不再需要处理。

**常见用户问题**：
- "我签到之后，为什么还没有取样计划？"→ 可能一卡通通知未成功到达，建议联系采样员手动创建。
- "取样计划显示未取样是什么意思？"→ 说明采样员还没有去现场采样。
- "取样计划已经作废了怎么办？"→ 需要重新创建新的取样计划。

### 6.2 样品

**用户视角含义**：
采样员从运料车上实际采集的那份原料，就是"样品"。在系统中，一个取样计划下面可以有多个样品（比如一车玉米可能取多个样品、不同位置的样品）。

每个样品都有样品编号，后续的化验都是围绕样品来进行的。

**系统中对应的数据表**：`op_sampling_plan_sample`

**样品的关键标识**：
- 样品类型为"原料"（sampling_type=3），区别于成品、库存、垫料。
- 样品编号（sample_no）是这个样品的唯一标识，可以用于追踪和查询。
- 合格状态（whether_qualified）：A1 表示合格，R1 表示不合格。

**常见用户问题**：
- "我的样品编号是什么？"→ 采样员在取样时会生成样品编号。
- "一个取样计划可以有多少个样品？"→ 取决于实际取样需求，可以取多个。
- "样品状态显示不合格是什么意思？"→ 化验结果中有至少一项超出了标准范围。

### 6.3 检测项目

**用户视角含义**：
对一份原料样品，需要检测哪些指标——比如水分含量、粗蛋白含量、黄曲霉毒素是否超标。每一个需要检测的指标就是一个"检测项目"。

不同原料的检测项目不同：玉米主要关注水分和毒素，豆粕主要关注蛋白含量。检测项目由系统根据标准配置自动确定。

**系统中对应的数据表**：`op_sampling_plan_item`

**检测项目的关键信息**：
- 项目名称（如"水分"、"粗蛋白"、"黄曲霉毒素B1"）。
- 检测结果（化验员录入的具体数值）。
- 上下限标准（什么样的结果算合格）。

**常见用户问题**：
- "这个样品需要检测哪些项目？"→ 由系统根据物料类型和牧场标准自动确定。
- "为什么我这个样品比别人多几个检测项目？"→ 不同物料类型的标准配置不同。
- "检测结果为空是什么意思？"→ 化验员还没录入这个项目的结果。

### 6.4 检测结果

**用户视角含义**：
化验员在实验室完成检测后，录入到系统中的具体数值（如"水分：14.5%"）或定性结论（如"黄曲霉毒素：阴性"），就是检测结果。一组完整的检测结果是判定合格与否的唯一依据。

**系统中对应的数据表**：
- 化验单主表（`op_test_result_base`）：一次化验操作的整体记录。
- 化验结果明细表（`op_test_result_info`）：每个样品每个检测项目的具体结果。

**化验单的状态流转**：
- **待提交（2）**：化验员正在录入结果，还没提交。
- **待审核（3）**：化验员已提交，等待审核员确认。
- **已审核（4）**：审核员已确认，结果是最终有效的。

**常见用户问题**：
- "化验结果什么时候出来？"→ 取决于化验员的工作安排和检测项目数量。
- "化验单待审核是什么意思？"→ 化验员已经提交了结果，审核员还没有确认。
- "化验单被退回了怎么办？"→ 审核员可能发现了数据问题，化验员需要检查并重新提交。

### 6.5 合格判定

**用户视角含义**：
将化验员录入的检测结果与预设的质量标准进行比对，判断这个检测项目是"合格"还是"不合格"。这个判定是在审核员点击"审核通过"时由系统自动完成的。

**判定依据**：系统预设的物料质量标准（存储在 `bs_invbill_item_standard` 表中），针对不同物料、不同牧场可能有不同的标准范围。

**判定结果类型**：

| 结果 | 含义 | 常见问题 |
|------|------|---------|
| A1（合格） | 检测结果在标准范围内 | 可以正常放行 |
| R1（不合格） | 检测结果超出标准范围 | 会生成不合格品记录，需要进一步处理 |
| A2（可接收） | 超出第一标准但在可接受范围内 | 【待确认】原料流程是否使用 |
| R2（不合格退货） | 严重超标，需要退货 | 【待确认】原料流程是否使用 |

**常见用户问题**：
- "为什么判我不合格？"→ 检测结果中有项目超出了标准范围，具体是哪个项目超标可以查看化验结果。
- "合格标准是什么？"→ 不同物料、不同牧场有不同的标准，需要查询具体配置。
- "可接收和合格有什么区别？"→ 【待确认】

### 6.6 放行状态

**用户视角含义**：
当一车原料的所有检测项目全部合格后，系统标记这车原料为"已放行"，意思是这车原料可以卸车入库、投入使用了。放行是原料进厂质检流程的最终目标。

**放行状态的值**：
- **未放行（0）**：还不满足放行条件——可能还没取样、还没化验完成、或者有不合格项。
- **已放行（1）**：所有检测合格，可以入库使用。

**谁可以查询放行状态**：
- 一卡通系统可以通过签到 ID 查询对应车辆的放行状态。
- LIMS 系统内可以查看取样计划的放行状态。

**常见用户问题**：
- "我的原料什么时候能放行？"→ 需要等所有检测项目完成且全部合格后才会放行。
- "为什么还没放行？"→ 可能原因：还没取样、化验未完成、结果未审核、或有不合格项。
- "谁负责放行？"→ 【待确认】是系统在全部合格后自动放行，还是需要人工点击放行。

## 7. 常见误解

### 误解1：司机签到就等于已经放行

**实际情况**：司机在一卡通系统签到，只是触发了原料取样计划的创建（告诉 LIMS 有一车新原料到了）。从签到到放行之间，还需要经过取样、化验录入、结果审核、合格判定等多个环节。签到只代表"开始处理"，放行才代表"处理完成、可以使用"。

### 误解2：有取样计划就等于已经取样

**实际情况**：取样计划的状态为"未取样（status=1）"时，表示计划已创建但采样员还没有去现场采样。只有状态变为"已取样（status=2）"时，才表示样品已经采集完毕。

### 误解3：已取样就等于已经化验完成

**实际情况**：取样只是把样品从车上采集下来。采集完成后，样品还需要送到化验室，由化验员排队检测、录入结果。从"已取样"到"化验完成"之间可能需要一定的工作时间。

### 误解4：化验完成不等于一定合格

**实际情况**：化验完成只是说明检测结果已经出来了，但结果可能合格也可能不合格。具体合格与否，要看每项检测结果是否在标准范围内。如果有一项不达标，整批原料就可能被判为不合格。

### 误解5：合格判定不等于一定已经同步到外部系统

**实际情况**：在 LIMS 系统中判定合格，只是在 LIMS 内部更新了状态。一卡通系统需要通过接口主动查询才能获取最新的放行状态。两个系统之间存在"查询时获取"的关系，不是实时推送。

### 误解6：AI 当前不能查询某辆车的实时放行状态

**实际情况**：AI 客服没有连接实时数据库的权限，无法直接查询"某辆车现在有没有放行"。AI 能告诉你的是：放行状态的含义、什么条件下会放行、如果没放行可能是什么原因。需要查询具体车辆状态，请通过一卡通系统或联系现场工作人员。

### 误解7：AI 当前不能自动放行

**实际情况**：放行操作涉及质量判定和审核确认，AI 客服不能代替人工进行放行操作。如果你需要放行，请确认所有化验结果已审核通过且全部合格。

### 误解8：原料流程不要和中心化验室委托单流程混淆

**实际情况**：这是两个完全不同的业务流程：
- **原料取样流程**：牧场/饲料厂自己检验自己进的原料，数据存在原料取样相关表中。
- **中心化验室委托单流程**：外部单位把样品送到检测中心委托检验，检测中心出具正式报告，数据存在委托单相关表中。

两种流程使用不同的系统模块、不同的数据表、不同的操作页面。如果你遇到的是"送样到检测中心"、"委托检验"、"检验报告"等场景，那属于中心化验室流程，不在本模块范围。

### 误解9：一车原料只需要取一个样品

**实际情况**：根据取样规范，一车原料可能需要取多个样品（不同位置、不同深度）。每个样品独立编号、独立化验，所有样品都合格才能放行。【待确认：实际的取样份数规则】

### 误解10：化验结果出来后不需要审核就能放行

**实际情况**：化验员录入结果后，化验单的状态是"待审核"。必须经过审核员确认（点击"审核通过"），结果才会生效。只有生效的结果才会参与合格判定和放行判断。

## 8. AI 客服回答边界

### AI 可以回答的问题

| 问题类型 | 示例 | 说明 |
|---------|------|------|
| **状态含义** | "未取样是什么意思？""已审核代表什么？" | 可以解释每个状态值的业务含义 |
| **操作流程** | "原料进厂后要经过哪些步骤？""谁负责取样？" | 可以说明完整的业务流程和各角色职责 |
| **为什么不能放行** | "为什么我的原料还没放行？" | 可以列举可能导致未放行的原因（未取样、化验中、待审核、有不合格项等） |
| **为什么还要等待** | "已经取样了为什么还没结果？" | 可以说明取样后还需要经过化验、审核等环节 |
| **下一步找谁** | "我应该联系谁处理？" | 可以根据当前状态建议联系采样员、化验员或审核员 |
| **概念区分** | "原料取样和中心化验室有什么区别？" | 可以帮助区分不同模块的业务范围 |
| **一卡通关系** | "签到之后会发生什么？" | 可以说明一卡通签到触发取样计划的机制 |
| **放行条件** | "放行需要满足什么条件？" | 可以说明全部检测合格才能放行的条件 |

### AI 不能回答的问题

| 问题类型 | 示例 | 原因 |
|---------|------|------|
| **实时状态查询** | "帮我查一下车牌沪A12345现在放行了没有？" | AI 不连接实时数据库，无法查询具体车辆的当前状态 |
| **实时检测结果** | "32号样品的粗蛋白含量是多少？" | AI 不连接实时数据库，无法查询具体检测数值 |
| **自动审核** | "帮我审核一下这个化验单。" | 审核需要人工判断，AI 不能代替 |
| **自动修改数据** | "帮我把这个结果改成合格。" | AI 不能修改系统中的任何数据 |
| **自动放行** | "帮我把这车原料放行。" | 放行需要确认检测结果，AI 不能代替操作 |
| **实时数据库查询** | "现在还有多少车原料没放行？" | AI 不能查询实时数据库或生成实时统计 |
| **中心化验室问题** | "我的委托单状态是什么？" | 中心化验室委托单属于另一个模块，不在本知识库范围 |
| **奶源质检问题** | "这批牛奶的质检结果出来了吗？" | 奶源质检使用独立系统，不在本模块范围 |
| **预测时间** | "化验什么时候能完成？" | 化验完成时间取决于化验员的工作安排，AI 无法预测 |
| **SAP/外部系统问题** | "SAP 系统里有没有收到化验结果？" | 外部系统集成不在 AI 当前覆盖范围 |
