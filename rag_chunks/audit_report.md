## 四、技术细节暴露检查（public chunk）

### 4.1 检查标准

visibility=public 的 chunk 不应暴露以下内容给非管理员角色：
- 数据库表名（如 `op_sampling_plan`）
- 字段名（如 `status`、`is_release`）
- 接口路径（如 `/ranch/plan/notifySignIn`）
- SQL / 数据库操作建议

### 4.2 发现的问题

| Chunk ID | 暴露内容 | 角色 | 严重度 |
|----------|---------|------|--------|
| 01-004 | 表名 `op_sampling_plan`、`op_sampling_plan_sample`、`op_sampling_plan_item`，字段名 `sampling_type` | all | 中 |
| 01-011 | 表名 `op_sampling_plan`，状态码 0/1/2/3 | all | 中 |
| 01-012 | 表名 `op_sampling_plan_sample`，字段名 `sample_no`、`sampling_type`、`whether_qualified` | sampler,tester | 中 |
| 01-013 | 表名 `op_test_result_base`、`op_test_result_info`、`bs_invbill_item_standard` | tester,reviewer | 中 |
| 01-014 | 表名引用，字段名 `is_release` | all | 中 |
| 01-019 | 表名 `op_sampling_plan.status`、`op_test_result_base.status` 等 | all | 中 |
| 03-003~006 | 每个状态 chunk 都暴露了 `op_sampling_plan.status` | all | 中 |
| 03-009~010 | 暴露了 `op_sampling_plan.is_release` | all | 中 |
| 03-011~012 | 暴露了 `op_sampling_plan_sample.whether_qualified` | all | 中 |
| 03-016~019 | 暴露了 `op_test_result_base.status`、`op_test_result_info` | tester,reviewer | 中 |

### 4.3 分析

这些 chunk 在设计时采取了"面向 all 角色但附带表名"的策略——即元数据中 role=all 或 role=sampler/tester，但正文中包含了表名和字段名。这在 RAG 检索时存在风险：司机、调度员提问时可能被检索到这些 chunk，从而暴露技术细节。

### 4.4 建议

**方案 A（推荐）**：不变更现有 chunk 内容，但在系统 Prompt（05-028）中强化"角色粒度过滤"规则——当检索到 role=all 的 chunk 且包含技术细节时，AI 应对非管理员用户自动剥离表名和字段名。

**方案 B**：将 role=all 且含技术细节的 chunk 拆分为两份：一份 role=driver/sampler/dispatcher（纯业务语言），一份 role=admin（含技术细节）。工作量大，本轮不推荐。

**采纳方案 A，优先级：建议修改（修改 05-028）。**

### 6.1 新口径要求

根据业务最新确认：
- 当前代码中**未找到** `is_release` 自动 set 值的方法。
- 前端**没有**放行字段。
- **禁止断言**：系统会自动放行、审核通过后会自动更新 is_release=1、用户可以手动点击放行、AI 可以确认某车已放行。
